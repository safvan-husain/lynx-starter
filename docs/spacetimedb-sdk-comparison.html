SpacetimeDB SDK Comparison

This compares the local Lynx port used by this app, the Dart package, and the official SpacetimeDB React integration from the TypeScript SDK. The last column focuses on gaps that matter if spacetimedb-lynx is expected to support full frontend application behavior.

Status after the latest Lynx work: the local package now has a functional React-style useTable hook backed by useSyncExternalStore, subscription error callbacks, and an iOS Lynx WebSocket transport using global native events. The counter UI no longer depends on REST SQL polling for its live counter value, and reducer actions now await SDK reducer acknowledgements directly instead of using an app-side timeout wrapper. Session continuity now reuses persisted tokens, validates restored auth sessions with a one-off query, and reconnects a dropped managed connection with the latest token.

Best fit today for controlled Lynx apps that need typed reducers plus live table subscriptions on the validated iOS path. Closer, not complete: React table hook, iOS transport, one-off query, reducer ack, and iOS session/reconnect gaps were reduced. Selected next candidate is reducer event callback / flag parity with the official SDK.

Validation Workflow

For changes that touch the SpacetimeDB module, transport, reducer calls, subscriptions, auth/session reads, or generated bundle behavior, the current validation path is:

  1. Restart or refresh the local in-memory SpacetimeDB server when the published database may be stale or belongs to a previous anonymous identity.
  2. Publish spacetimedb-counter to lynx-counter with anonymous local credentials.
  3. Run the Lynx app build and sync the generated bundle into the iOS, Android, and Harmony host assets.
  4. Build, install, and relaunch the iOS simulator host app.
  5. Validate the app UI against the live database, then confirm state through spacetime sql.
  6. Scan the simulator host log for reducer, SQL, subscription, and transport errors before marking the activity as validated.
